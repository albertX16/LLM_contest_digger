# 规则到当前系统边界的参考映射

> 本文是辅助参考，不是另一份任务简报。完整流程、人员分工和待定方案以 [README.md](../README.md) 为准。

## 1. 本文的作用

本文只说明目前哪些系统边界直接来自比赛规则，哪些内容是项目内部选择，避免把备选方案误写成官方要求。

## 2. 可以直接映射的规则

| 规则或约束 | 对系统的直接影响 |
|---|---|
| 最终提交是单因子 | 每个候选最终都要能够生成 `date / instrument / factor` |
| 需要可执行代码 | 抽象 Token 不能直接提交，需要转化为可执行因子程序 |
| 禁止未来数据 | Token 检查和因子执行需要控制历史窗口和数据时点 |
| 禁止外部数据 | 论文可以提供思路，但不能引入额外数值数据 |
| 后台不能访问互联网 | 最终提交不能依赖在线模型、在线论文库或远程接口 |
| 运行时间有限 | 代码需要考虑计算成本，但具体分层和缓存策略由项目决定 |
| 最多提交 50 个因子 | 自动因子和传统因子最终需要统一统筹 |
| 评分含单因子和组合贡献 | 单因子质量与后期互补性都需要进入研究视野 |

## 3. 当前系统的工作边界

### 人员一模块

负责最外层输入和生成端大语言模型，包括规则、分类因子库、算子目录、历史反馈以及可选论文材料。

### 人员二模块

负责遗传搜索、具体 Token、算子调用、合法性检查和 Token 到可执行因子程序的转换。

人员二交付计算能力，不提前生成全部候选的全年 DataFrame。

### 人员三模块

负责按短周期和长周期调用因子程序、回测、筛选、结果解释和轮次总结。

### 人员四模块

负责总体框架、接口衔接和调试。人员一、二、三在代码上通过人员四连接。

### 人员五工作

负责传统因子、研究资料和两个赛道的整体统筹。具体工作形式由团队后续继续讨论。

## 4. 当前接口方向

```text
RoundContext
    ↓
人员一：GenerationPlan
    ↓
人员二：CandidateBundle（包含可执行因子程序）
    ↓
人员三：EvaluationBundle
    ↓
人员四整理为下一轮 RoundContext
```

这些名称是当前工作接口，不代表字段已经最终冻结。

## 5. 不应从规则直接推出的结论

比赛规则本身没有替项目决定：

- 是否使用论文驱动生成；
- 一轮运行多少代遗传搜索；
- 大语言模型是否负责主要筛选；
- 是否采用强化学习；
- 因子相似度如何计算；
- 短周期筛选阈值；
- 候选因子库门槛；
- 人员的开发阶段和验收标准；
- 生成端和评估端是否共享记忆。

这些内容应作为待选方案，由相应人员查阅资料、开发接口并结合实际结果继续讨论。

## 6. 当前仍需核验的规则接口问题

- 研发测试端口是否自动完成行业和 BARRA 风格剥离；
- 研发端口和最终平台的处理是否一致；
- 官方主函数的准确签名；
- 缺失率限制的准确口径；
- 最终提交的文件组织和运行方式。
